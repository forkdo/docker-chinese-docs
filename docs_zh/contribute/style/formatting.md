## 标题和副标题

读者对标题、列表和链接的关注度更高，因此确保这些内容的前两到三个词尽可能“前置”关键信息非常重要。

### 最佳实践

- 标题和副标题应让读者清楚地知道页面内容。
- 它们应简洁、准确地描述内容主题。
- 标题应简短（不超过八个词），直截了当，使用通俗、主动的语言。
- 应避免使用双关语、俏皮话和文化引用。
- 省略开头的冠词（a, the 等）。

## 页面标题

页面标题应以动作为导向。例如：
- _启用 SCIM_
- _安装 Docker Desktop_

### 最佳实践

- 确保页面标题与目录（TOC）条目一致。
- 如果要在目录文件（\_toc.yaml）中使用冒号（:），必须用双引号（""）将整个标题括起来，以避免破坏构建。
- 如果向 TOC 文件添加新条目，确保其以斜杠（/）结尾。否则，页面将不显示侧边导航。

## 图片

图片（包括截图）可以帮助读者更好地理解概念。但应谨慎使用，因为它们容易过时。

### 最佳实践

- 截图时：
  - 不要使用 lorem ipsum 文本。尝试模拟用户在真实场景中使用该功能的情况，并使用真实的文本。
  - 仅捕获相关的用户界面。不要包含不必要的空白或与说明无关的界面区域。
  - 保持图片尺寸小。如果不需要显示整个屏幕宽度，就不要这样做。
  - 检查图片在页面上的渲染效果。确保图片不模糊或过于突出。
  - 保持一致性。协调页面上已有的截图，提供一致的阅读体验。
  - 为减轻 Git 仓库的负担，请压缩图片（无损压缩）。务必在将图片添加到仓库之前进行压缩。压缩图片后再添加到仓库实际上会加剧对 Git 仓库的影响（不过，浏览时仍能优化带宽）。
  - 别忘了从仓库中删除不再使用的图片。

有关如何在内容中添加图片的信息，请参阅 [Useful component and formatting examples](../components/images.md)。

## 链接

谨慎添加过多链接，尤其是在正文内容中。过多的链接会分散注意力或让用户离开当前页面。

当用户跟随链接时，他们常常会失去思路，无法返回到原始页面，尽管他们尚未完全理解页面上的所有信息。

最佳的链接为读者提供了另一种浏览信息的方式。

### 最佳实践

- 使用不会误导或过度承诺的纯文本。
- 简短且具有描述性（大约五个词最佳）。
- 让用户能够（在相当程度上）预测选择链接后会得到什么。尽可能在链接中使用目标页面的标题文本。
- 在链接开头前置用户和动作导向的术语（例如：下载我们的应用）。
- 避免使用通用的行动号召（如点击此处、了解更多信息）。
- 添加超链接文本时不要包含结尾标点符号（例如句号、问号或感叹号）。
- 不要将链接文本设为斜体或粗体，除非它在正常正文中也是如此。

有关如何在内容中添加链接的信息，请参阅 [Useful component and formatting examples](../components/links.md)。

## 代码和代码示例

将以下内容格式化为代码：Docker 命令、说明和文件名（包名）。

要应用内联代码样式，请用单个反引号（`）包裹文本。

有关如何在内容中添加代码块的信息，请参阅 [Useful component and formatting examples](../components/code-blocks.md)。

### 命令的最佳实践

- 命令提示符和 shell：
  - 对于非特权 shell，在代码块中的命令前加上 $ 提示符。
  - 对于特权 shell，在代码块中的命令前加上 # 提示符。
  - 对于远程 shell，在命令前添加远程机器的上下文并省略路径。例如，`user@host $`。
  - 添加任何特定于 Windows 的命令时，请指定 Windows shell（命令提示符、PowerShell 或 Git Bash）。不需要为每个 Windows shell 都包含命令。
  - 在添加多个操作系统或 shell 的命令时使用标签页。有关如何在内容中添加标签页的信息，请参阅 [Tabs](../components/tabs.md)。
- 用户将复制和运行的命令：
  - 如果命令产生输出或需要用户验证结果，请在代码块中添加单个命令。
  - 将命令输出添加到与命令分开的代码块中。
- 用户不会复制和运行的命令：
  - 使用与 POSIX 兼容的语法。不需要包含 Windows 语法。
  - 将可选参数用方括号（ [ ] ）括起来。
  - 在互斥参数之间使用管道符（ \| ）。
  - 在重复参数后使用三个点（ ... ）。

### 代码的最佳实践

- 当代码块嵌套在列表项下时，将代码块缩进 3 个空格。
- 省略代码时使用三个点（ ... ）。

## 提示框

使用提示框来强调页面上的选定信息。

### 最佳实践

- 将单词 Warning、Important 或 Note 格式化为粗体。提示框内的内容不要加粗。
- 避免在一页上放置过多的文本提示框。如果使用过多，它们会显得杂乱，并削弱其影响。

| 文本提示框 | 使用场景                                                                                                                                                                   | 颜色或提示框 |
| ------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------- |
| Warning      | 使用 Warning 标签向读者发出信号，表明某些操作可能导致硬件损坏或软件数据丢失。                                                                      | 红色                  |
|              | ✅ 示例：Warning: 当您使用 docker login 命令时，您的凭据会以 base64 编码形式存储在主目录的 .docker/config.json 文件中。 |                      |
| Important    | 使用 Important 标签向读者发出信号，表明某些操作可能导致较低程度的问题。                                                                                   | 黄色               |
|              | ✅ 示例：更新 Docker Desktop 条款                                                                                                                                      |                      |
| Note         | 使用 Note 标签表示可能不适用于所有读者的信息，或与主题相关的附加信息。                                                                 | 蓝色                 |
|              | ✅ 示例：删除仓库会删除其中包含的所有镜像及其构建设置。此操作无法撤销。                                                          |                      |

有关如何在内容中添加提示框的信息，请参阅 [Useful component and formatting examples](../components/call-outs.md)。

## 导航

在记录如何在 Docker Desktop 或 Docker Hub UI 中导航时：

- 始终使用“位置，然后动作”。例如：
  _从 **Visibility** 下拉列表（位置）中选择 Public（动作）。_
- 简洁且具体。例如：
  做：_选择 **Save**。_
  不做：_选择 **Save** 以使更改生效。_
- 如果步骤必须包含原因，以原因开头。这有助于用户更快地扫描。
  做：_要在合并请求中查看更改，请选择链接。_
  不做：_选择合并请求中的链接以查看更改_

## 可选步骤

如果步骤是可选的，以单词 Optional 开头，后跟句号。

例如：

_1. Optional. 输入作业的描述。_

## 占位符文本

您可能需要提供使用特定值的命令或配置。

在这种情况下，使用 < 和 > 来标出读者必须用自己的值替换文本的位置。例如：

`docker extension install <name-of-your-extension>`

## 表格

使用表格以简单明了的方式描述复杂信息。

请注意，在许多情况下，无序列表足以描述每个项目的单一简单描述。但是，如果您有最适合用矩阵描述的数据，表格是最佳选择。

### 最佳实践

- 表格标题使用句子大小写。
- 为保持表格的可访问性和可扫描性，表格不应有任何空单元格。如果某个单元格没有其他有意义的值，请考虑输入 N/A（不适用）或 None。

有关如何在内容中添加表格的信息，请参阅 [Useful component and formatting examples](../components/tables.md)。

## 引用文件类型

当讨论文件类型时，使用该类型的正式名称。不要使用文件扩展名来泛指文件类型。

    | 正确 | 错误 |
    | --- | --- |
    | a PNG file | a .png file |
    | a Bash file | an .sh file |

## 引用单位

引用测量单位时，使用全大写的缩写形式，数值和单位之间用空格分隔。例如：

    | 正确 | 错误 |
    | --- | --- |
    | 10 GB | 10GB |
    | 10 GB | 10 gb |
    | 10 GB | 10 gigabytes |
