## 语音

在 Docker，我们始终站在用户的角度思考。我们是为开发者服务的开发者。我们以经验和知识说话，不带任何傲慢或自负。我们希望在不造成困惑或施加压力的情况下，为用户提供信息并赋能。

我们不介意用一点俏皮的幽默来活跃对话气氛（因为我们并不把自己看得太重），但我们始终尊重他人。我们以清晰、同理心和机智进行沟通；我们所说的每一句话都应具有信息性和鼓励性。

我们的语气和内容与我们的价值观保持一致。写作时最重要的原则是遵循“4C”：

1. **正确 (Correct)**
2. **简洁 (Concise)**
3. **完整 (Complete)**
4. **清晰 (Clear)**

我们确保信息准确、简洁、全面且易于理解。我们尽可能简化句子，但包含足够的细节，以便用户完成预期任务。

这意味着，当我们编写文档和用户界面文案时：

1. **我们诚实可信。** 我们提供所有事实，不使用误导或模棱两可的陈述。我们并非总是拥有所有答案，但我们竭尽全力让开发者的生活更美好，并会如实告知进展。
2. **我们言简意赅。** 我们理解用户所处的复杂和冗长信息的行业，因为我们来自同一世界。Docker 不会用空洞的词语或复杂的隐喻来堆砌沟通内容。我们清晰明了，直击要点。
3. **我们轻松自在。** 我们的风格随意但不懒散，聪明但不傲慢，专注但不冷漠。我们的声音应具有亲和力和温暖感。
4. **我们包容开放。** 无论编写过多少代码，开发者就是开发者。每个人都是我们社区的重要组成部分，无论其行业或经验水平如何。我们接纳来自各行各业、各种经验水平的开发者。

## 语调

Docker 的语调通常是非正式的，但我们认为清晰比幽默更重要。我们很放松，但不会不当或不专业。

### 友好的语调

使用自然、友好且尊重的语调，避免过于口语化或充满行话。写作的目的是为开发者提供信息并赋能，而不是造成困惑或施加压力。使用缩略形式是可以的，只要句子不会变得过于俚语化或随意。

**避免过度礼貌。** 友好和礼貌是好的，但在技术文档或用户界面文案中使用“请”可能显得过于客套。

### 积极的语言

使用积极的语言。与其强调限制和用户无法做的事情，不如强调积极的结果。

例如，**不要说**：

“*单点登录 (SSO)、镜像访问管理、注册表访问管理等功能在 Docker Team 订阅中不可用。*”

**而应该说**：

“*单点登录 (SSO)、镜像访问管理、注册表访问管理等功能在 Docker Business 订阅中可用。*”
