本指南提供了编写教程式使用指南的说明和最佳实践，帮助用户使用 Docker 实现特定目标。这些指南将与我们的产品手册和参考材料一起，展示在网站的 [指南部分](/guides/_index.md)。

我们的文档使用 Markdown 编写，元数据使用 YAML 前置内容。我们使用 [Hugo](https://gohugo.io) 构建网站。有关为 Docker 文档编写内容的更多信息，请参考我们的 [CONTRIBUTING.md](https://github.com/docker/docs/tree/main/CONTRIBUTING.md)。

## 指南的目的

我们的使用指南旨在：

- 教育用户如何在各种场景中使用 Docker。
- 通过逐步教程提供实践、动手的经验。
- 帮助用户实现与其兴趣或项目相关的特定目标。

## 受众

- 经验水平：从初学者到高级用户。
- 角色：开发者、系统管理员、DevOps 工程师等。
- 技术：各种编程语言和框架。

## 指南的元数据

每个指南必须在文档开头包含元数据部分。这些元数据帮助用户根据兴趣和需求发现和筛选指南。

### 元数据格式示例

```yaml
---
title: 使用 Docker 部署机器学习模型
linkTitle: Docker 用于 ML 部署
description: 了解如何使用 Docker 容器化和部署机器学习模型。
summary: |
  本指南将引导您完成容器化机器学习模型并使用 Docker 部署的步骤，
  实现可扩展且便携的 AI 解决方案。
tags: [machine-learning, deployment]
languages: [python]
params:
  time: 30 分钟
---
```

### 元数据字段

- `title`（必需）：指南的主要标题，使用句子大小写。
- `linkTitle`（可选）：导航菜单中使用的较短标题。
- `description`（必需）：用于 SEO 的简明描述。
- `summary`（必需）：指南内容的简要概述。
- `languages` \*（可选）：使用的编程语言列表。
- `tags` \*（可选）：涵盖的领域或主题。
- `params`
  - `time`（可选）：阅读或完成的预估时间。

\* 至少应用 `languages` 或 `tags` 分类之一。这些值用于将页面与指南着陆页上的筛选选项关联。

## 文档结构

我们的指南强调通过实践学习。根据指南类型，结构可能有所不同，以最好地适应内容并提供流畅的学习体验。

所有指南都直接位于 Docker 文档仓库的 `content/guides/` 目录下。指南可以是单页或多页。对于多页指南，每页都是顺序工作流中的一个步骤。

如果您要创建单页指南，在指南目录中创建一个 Markdown 文件：

```bash
# 创建文件
touch content/guides/my-docker-guide.md
# 或者如果您安装了 Hugo：
hugo new content/guides/my-docker-guide.md
```

要创建多页指南，创建一个目录，其中每页都是一个 Markdown 文件，并在 `_index.md` 文件中包含指南的介绍。

```bash
# 为指南创建索引页
mkdir content/guides/my-docker-guide.md
touch content/guides/my-docker-guide/_index.md
# 或者如果您安装了 Hugo：
hugo new content/guides/my-docker-guide/_index.md
```

然后根据需要在 `content/guides/<dir>/<page>.md` 下创建指南页面。（元数据位于 `_index.md` 页面上，您不需要为单个页面分配元数据。）

### 特定框架或语言的指南

对于演示如何将 Docker 与特定框架或编程语言一起使用的指南，考虑以下大纲：

1. **介绍**
   - 简要介绍 Docker 上下文中的框架或语言。
   - 解释用户在指南结束时将实现的目标。
   - 列出所需的软件、工具和知识。
2. **开发环境设置**
   - 指导用户设置开发环境。
   - 包含编写或获取示例代码的说明。
   - 展示如何为本地开发运行容器。
3. **构建应用程序**
   - 解释如何为应用程序创建量身定制的 Dockerfile。
   - 提供构建 Docker 镜像的逐步说明。
   - 如果适用，展示如何使用 Docker 测试应用程序。
4. **使用 Docker 部署**
   - 展示如何在 Docker 容器中运行应用程序。
   - 讨论配置选项和最佳实践。
5. **最佳实践和结论**
   - 提供针对框架或语言优化 Docker 使用的技巧。
   - 总结关键要点，建议下一步和进一步阅读。

### 用例指南

对于专注于使用 Docker 完成特定目标或用例的指南（例如，部署机器学习模型），使用灵活的大纲以确保逻辑流程。

以下大纲是一个示例。结构应根据内容进行调整，以确保清晰、合乎逻辑的进展。根据您的用例指南的主题，确切的结构会有所不同。

1. **介绍**
   - 描述问题或目标。
   - 解释在此上下文中使用 Docker 的好处。
2. **先决条件**
   - 列出所需的工具、技术和先验知识。
3. **设置**
   - 提供设置环境的说明。
   - 包含任何必要的配置步骤。
4. **实现**
   - 逐步引导流程。
   - 使用代码片段和说明说明要点。
5. **运行或部署应用程序**
   - 指导用户如何执行或部署解决方案。
   - 讨论任何验证步骤以确保成功。
6. **结论**
   - 回顾实现的目标。
   - 建议进一步阅读或高级主题。

## 示例代码

如果您创建示例仓库以提供与指南配套的源代码，我们强烈建议您将该仓库发布在 GitHub 上的 [dock样本](https://github.com/dockersamples) 组织下。将源代码发布在此组织下，而不是您的个人账户下，可确保文档网站维护者在发布后仍能访问源代码。如果指南中出现错误或问题，文档团队可以更容易地更新指南及其相应的示例仓库。

将示例托管在官方样本命名空间下也有助于建立用户对指南的信任。

### 在 `dock样本` 下发布仓库

要在 `dock样本` 组织下发布您的仓库，请使用 [dock样本模板](https://github.com/dockersamples/sample-app-template) 在您的个人命名空间下启动样本仓库。当您完成内容草稿并为文档打开拉取请求时，我们可以将仓库转移到 dock样本 组织。

## 写作风格

所有标题和标题使用 [句子大小写](/contribute/style/formatting.md#capitalization)。这意味着只有第一个单词和专有名词大写。

### 语气和语调

- **清晰和简洁**：使用简单语言和短句有效传达信息。
- **主动语态**：使用主动语态吸引读者（例如，“运行命令”而不是“应该运行命令”）。
- **一致性**：在整个指南中保持一致的语气和术语。

详细指南，请参考我们的 [语气和语调文档](/contribute/style/voice-tone.md)。

### 格式约定

- **标题**：H2 用于主要部分，H3 用于子部分，遵循句子大小写。
- **代码示例**：提供完整的、可工作的代码片段，并带有语法高亮。
- **列表和步骤**：对顺序步骤使用编号列表，对非顺序信息使用项目符号。
- **强调**：对 UI 元素使用粗体（例如，**按钮**），对强调使用斜体。
- **链接**：使用描述性链接文本（例如，[安装 Docker](/get-started/get-docker.md)）。

更多详细信息，请参阅我们的 [内容格式指南](/contribute/style/formatting.md) 和 [语法和风格规则](/contribute/style/grammar.md)。

## 最佳实践

- **测试所有说明**
  - 确保所有代码和命令按预期工作。
  - 验证指南可以从头到尾成功遵循。
- **相关性**
  - 专注于实际应用和场景。
  - 保持内容与最新 Docker 版本同步。
- **故障排除提示**
  - 预见常见问题并提供解决方案或参考。
- **视觉辅助**
  - 在增强理解的地方包含屏幕截图或图表。
  - 添加说明和替代文本以提高可访问性。
- **第三方工具**
  - 避免要求用户安装第三方工具或修改其本地开发环境。
  - 在适用时优先使用容器化工具和方法（例如 `docker exec`）。
  - 某些工具可以合理地假设为已安装或指南的先决条件，例如 Node.js 和 npm。请根据您的判断决定。

## 提交流程

- **提案**
  - 在 [Docker 文档 GitHub 仓库](https://github.com/docker/docs) 上提出问题，请求添加新指南。
  - 一旦提案被接受，通过分叉仓库并为您的工作创建分支开始编写指南。

    > [!NOTE]
    > 避免从您分叉的 `main` 分支贡献，因为这会使维护者更难帮助您修复可能出现的问题。

- **审查**
  - 为语法、清晰度和遵循指南进行校对您的指南。
  - 一旦草稿准备就绪，提出拉取请求，并引用原始问题。
  - Docker 文档团队和主题专家将审查您的提交，并在拉取请求中直接提供反馈。
- **发布**
  - 当审查者批准并合并您的拉取请求后，您的指南将发布在 Docker 文档网站上。

感谢您为 Docker 社区做出贡献。您的专业知识帮助全球用户利用 Docker 的力量。
