通过正确的配置，您可以确保开发人员轻松访问必要的资源，同时防止未授权访问。本文档将指导您识别 Docker 用户，以便在 Docker 组织内高效分配订阅席位，并分配角色以与组织结构保持一致。

## 识别您的 Docker 用户和账户

在设置角色和权限之前，重要的是清楚了解组织中谁需要 Docker 访问权限。重点关注收集活跃用户的全面视图、他们在项目中的角色以及他们与 Docker 资源的交互方式。此过程可以通过设备管理软件或手动评估来支持。鼓励所有用户更新其 Docker 账户，使用组织邮箱地址，确保与您的订阅无缝集成。

有关如何执行此操作的步骤，请参阅 [组织入职的步骤 1](/manuals/admin/organization/onboard.md)。

## 战略性分配角色

邀请成员加入您的 Docker 组织时，您需要为其分配一个角色。

Docker 的预定义角色为各种组织需求提供了灵活性。有效分配角色可确保可访问性和安全性之间的平衡。

- 成员：非管理角色。成员可以查看同一组织中的其他成员。
- 编辑者：对组织具有部分管理访问权限。编辑者可以创建、编辑和删除仓库。他们还可以编辑现有团队的访问权限。
- 所有者：对组织具有完全管理访问权限。所有者可以管理组织仓库、团队、成员、设置和账单。

更多信息，请参阅 [角色和权限](/manuals/enterprise/security/roles-and-permissions.md)。

### 通过团队增强管理

Docker 中的团队提供了一种结构化的方式来管理成员访问，并提供额外的权限级别。它们简化了权限管理，并实现策略的一致应用。

- 将用户组织到与项目、部门或职能角色对齐的团队中。这种方法有助于简化资源分配，并确保访问控制的清晰性。
- 在团队级别而非个人级别分配权限。例如，开发团队可能对某些仓库具有“读写”访问权限，而 QA 团队具有“只读”访问权限。
- 随着团队增长或职责变化，您可以轻松更新权限或添加新成员，保持一致性而无需重新配置个人设置。

更多信息，请参阅 [创建和管理团队](/manuals/admin/organization/manage-a-team.md)。

### 示例场景

- 开发团队：为开发人员分配成员角色，授予他们对编码和测试所需仓库的访问权限。
- 团队负责人：为团队负责人分配编辑者角色，以便在其团队内管理资源和控制仓库。
- 组织监督：将组织所有者或公司所有者角色限制为少数负责账单和安全设置的受信任人员。

### 最佳实践

- 应用最小权限原则。仅向用户分配其角色所需的最低权限。
- 定期审查角色分配，确保它们与不断发展的团队结构和组织职责保持一致。
