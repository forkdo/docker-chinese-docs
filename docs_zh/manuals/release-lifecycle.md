本文档详细介绍了 Docker 的产品发布生命周期，以及 Docker 如何定义每个阶段。同时提供有关产品淘汰流程的信息。功能和产品可能会经历以下部分或全部阶段。

## 生命周期阶段

| 生命周期阶段  | 客户可用性 | 支持可用性 | 限制 | 淘汰 |
| --- | --- | ---- | ---| ---|
|[实验性 (Experimental)](#experimental)| 有限可用 | 社区支持 | 软件可能存在限制、错误和/或稳定性问题 | 可随时终止，不另行通知 |
|[Beta](#beta) | 所有用户或参与 Beta 反馈计划的用户 | 社区支持 | 软件可能存在限制、错误和/或稳定性问题 | 可随时终止，不另行通知 |
| [早期访问 (EA)](#early-access-ea) | 所有用户或参与早期访问反馈计划的用户 | 完整支持 | 软件可能存在限制、错误和/或稳定性问题，这些限制将被记录 | 遵循 [淘汰流程](#retirement-process) |
| [正式发布 (GA)](#general-availability-ga) | 所有用户 | 完整支持 | 受支持用例的限制很少或没有 | 遵循 [淘汰流程](#retirement-process) |

### 实验性 (Experimental)

实验性功能是 Docker 当前正在试验的功能。访问实验性功能的客户有机会测试、验证并反馈未来功能，这有助于我们将精力集中在为客户提供最大价值的方向上。

**客户可用性：** 实验性功能的可用性有限。部分用户可能可以访问零个、一个或多个实验性功能。

**支持：** 实验性功能的支持通过社区支持渠道和论坛尽力提供。

**限制：** 实验性功能可能存在显著限制，如功能限制、性能限制和 API 限制。功能和程序接口可能随时更改，恕不另行通知。

**淘汰：** 在实验期间，Docker 将决定是否继续推进该功能的生命周期。我们保留在任何时候更改范围或终止实验性产品或功能的权利，恕不另行通知，具体如我们的 [订阅服务协议](https://www.docker.com/legal/docker-subscription-service-agreement) 所述。

### Beta

Beta 版本是潜在未来产品或功能的首次发布。参与我们 Beta 计划的客户有机会测试、验证并反馈未来功能，这有助于我们将精力集中在为客户提供最大价值的方向上。

**客户可用性：** Beta 版本的参与通过邀请或使用产品中明确标识的 Beta 功能。Beta 邀请可能是公开或私密的。

**支持：** Beta 功能的支持通过社区支持渠道和论坛尽力提供。

**限制：** Beta 版本可能存在显著限制，如功能限制、性能限制和 API 限制。功能和程序接口可能随时更改，恕不另行通知。

**淘汰：** 在 Beta 期间，Docker 将决定是否继续推进该功能的生命周期。我们保留在任何时候更改范围或终止 Beta 产品或功能的权利，恕不另行通知，具体如我们的 [订阅服务协议](https://www.docker.com/legal/docker-subscription-service-agreement) 所述。

### 早期访问 (EA)

早期访问功能是可能存在潜在功能限制的产品或功能，作为增量发布策略的一部分，面向特定用户组启用。它们已准备好面向全球发布，只需进行一些微调。

**客户可用性：** 早期访问功能可以向所有客户或特定用户群体发布，作为现有功能的补充或替代。

**支持：** 早期访问功能提供的支持级别与正式发布 (GA) 功能和产品相同。

**限制：** 早期访问版本可能存在显著限制，如功能限制、性能限制和 API 限制，尽管这些限制将被记录。功能和程序接口的破坏性更改将遵循下文所述的 [淘汰流程](#retirement-process)。

**淘汰：** 如果我们在正式发布之前淘汰早期访问产品，我们将努力遵循下文所述的 [淘汰流程](#retirement-process)。

### 正式发布 (GA)

正式发布功能是完全可用的产品或功能，所有 Docker 客户均可公开访问。

**客户可用性：** 所有 Docker 用户均可根据其订阅级别访问 GA 功能。

**限制：** 正式发布功能和产品在受支持用例方面限制很少或没有。

**支持：** 所有 GA 功能均提供完整支持，详情请参阅我们的 [支持页面](https://www.docker.com/support/)。

**淘汰：** 正式发布功能遵循下文所述的 [淘汰流程](#retirement-process)。

## 淘汰流程

淘汰或弃用功能的决定遵循严格的流程，包括了解需求、使用情况、功能淘汰的影响，以及最重要的客户反馈。我们的目标是将资源投入到能为最多客户增加最大价值的领域。

Docker 致力于与客户进行清晰、透明和主动的互动，尤其是在平台变更方面。为此，我们在淘汰功能时将尽力遵循以下准则：

- **提前通知：** 对于主要功能或产品的淘汰，我们将尝试至少提前 6 个月通知客户。
- **可行的替代方案：** Docker 将努力在淘汰功能时为客户提供可行的替代方案。这些替代方案可能来自 Docker 的其他产品，或来自第三方提供商的推荐方案。在可能且合适的情况下，Docker 将自动将客户迁移到已淘汰功能的替代方案。
- **持续支持：** Docker 承诺在功能淘汰日期之前持续提供支持。

在特殊情况下，我们可能需要加速功能的淘汰时间表，例如为保护平台完整性或客户及其他人的安全而必须进行的关键更改。在这种情况下，这些更改应尽可能快速发生。

同样，集成的第三方软件或服务可能需要因第三方决定更改或淘汰其解决方案而被淘汰。在这种情况下，淘汰的节奏将不受我们控制。

然而，即使在这些情况下，我们也会尽可能提前通知。
