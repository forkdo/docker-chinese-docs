Docker 的网络子系统是可插拔的，使用驱动来实现。默认提供了多种驱动，提供核心网络功能：

- `bridge`：默认的网络驱动。如果你未指定驱动，这就是你正在创建的网络类型。当你的应用在需要与同一主机上的其他容器通信的容器中运行时，通常使用桥接网络。
  参见 [Bridge 网络驱动](bridge.md)。

- `host`：移除容器与 Docker 主机之间的网络隔离，直接使用主机的网络。
  参见 [Host 网络驱动](host.md)。

- `overlay`：覆盖网络将多个 Docker 守护进程连接在一起，使 Swarm 服务和容器能够跨节点通信。这种策略消除了在操作系统级别进行路由的需要。
  参见 [Overlay 网络驱动](overlay.md)。

- `ipvlan`：IPvlan 网络让用户完全控制 IPv4 和 IPv6 地址。VLAN 驱动在此基础上提供对第 2 层 VLAN 标记的完全控制，甚至为希望集成底层网络的用户提供 IPvlan L3 路由。
  参见 [IPvlan 网络驱动](ipvlan.md)。

- `macvlan`：Macvlan 网络允许你为容器分配一个 MAC 地址，使其在网络中表现为一个物理设备。Docker 守护进程通过 MAC 地址将流量路由到容器。当你处理需要直接连接到物理网络（而不是通过 Docker 主机的网络栈路由）的遗留应用程序时，使用 `macvlan` 驱动有时是最佳选择。
  参见 [Macvlan 网络驱动](macvlan.md)。

- `none`：完全隔离容器与主机及其他容器。`none` 不适用于 Swarm 服务。
  参见 [None 网络驱动](none.md)。

- [网络插件](/engine/extend/plugins_network/)：你可以安装并使用第三方网络插件与 Docker 配合。

### 网络驱动总结

- 默认的桥接网络适用于运行不需要特殊网络功能的容器。
- 用户定义的桥接网络使同一 Docker 主机上的容器能够相互通信。用户定义的网络通常为属于同一项目或组件的多个容器定义一个隔离的网络。
- Host 网络与容器共享主机的网络。使用此驱动时，容器的网络与主机不隔离。
- 当你需要在不同 Docker 主机上运行的容器之间通信，或多个应用使用 Swarm 服务协同工作时，覆盖网络是最佳选择。
- 当你从虚拟机设置迁移，或需要容器在网络中表现为物理主机（每个都有唯一的 MAC 地址）时，Macvlan 网络是最佳选择。
- IPvlan 与 Macvlan 类似，但不为容器分配唯一的 MAC 地址。当网络接口或端口可分配的 MAC 地址数量受限时，考虑使用 IPvlan。
- 第三方网络插件允许你将 Docker 与专用网络栈集成。
